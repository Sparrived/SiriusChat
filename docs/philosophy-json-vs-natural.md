# 技术讨论：JSON / think+say / 自然语言输出的人格化差异

> **背景**：用户此前与 Claude 讨论过"主模型用 JSON 输出是否更好"，Claude 认为 JSON 会导致输出工程化。用户想知道 think+say 标签是否也有同样问题。
>
> **记录时间**：2026-04-18

## 问题的本质

这不是"格式问题"，而是**"框架化程度"问题**。

模型在被要求输出结构化内容时，会不自觉地进入"填写表单"模式——它会先思考"这个字段应该填什么"，而不是"我现在是什么感受"。这和人类被要求在表格里写"今日心情"时，写出的东西和日记里写的完全不一样的道理一样。

## 三种格式的对比

### 1. JSON 输出（最工程化）

```json
{
  "inner_thought": "Alice 平时很少说私事的，这次肯定真的很累",
  "reply": "摸摸头，需要我给你点外卖吗",
  "emotion": {"valence": -0.3, "arousal": 0.2}
}
```

**为什么容易工程化**：
- `"inner_thought"` 这个键名本身就在说"请写一个'内心想法'"，模型会把它当作一个**字段来填写**
- `"reply"` 键名暗示"这是给用户的正式回复"，模型会倾向于写得更"正确"、更"安全"
- JSON 语法约束（引号、逗号、括号）让模型在生成时持续进行**语法检查**，分散了对"角色体验"的注意力
- 最致命的是：**inner_thought 和 reply 被明确分开了**。模型知道 inner_thought 不会发给用户，所以它可能会写得更"诚实"，但这种"诚实"是**被框架规训过的诚实**——像面试时的自我介绍，知道对方在看所以刻意表现真诚

**Claude 的判断是对的**：JSON 会让模型进入"系统提示词执行模式"而不是"角色体验模式"。

### 2. think+say 标签（中等工程化）

```
<think>Alice 平时很少说私事的，这次肯定真的很累</think>
<say>摸摸头，需要我给你点外卖吗</say>
```

**比 JSON 好的地方**：
- 没有键名约束，`<think>` 和 `<say>` 只是**分隔符**，不是**字段定义**
- 标签内部是纯自然语言，模型不需要考虑语法格式
- 模型可以在 think 和 say 之间自由过渡，而不是"填完 A 字段再填 B 字段"

**但仍然有的问题**：
- 模型知道 `<think>` 里的内容不会发给用户，所以可能写得更"直白"，但这种"直白"可能过于**功能性**（"用户在求助，我应该安慰"）而不是**体验性**（"看到 Alice 这样我心里一沉"）
- 两个标签的**结构暗示**仍然存在：模型可能会把 think 当作"分析"，say 当作"执行"，导致 think 变成**角色分析报告**而不是**角色内心活动**

### 3. 完全自然语言 + 后处理分离（最自然）

```
（看到 Alice 说这句话，我心里一沉。她平时很少说私事的，这次肯定真的很累。）
摸摸头，需要我给你点外卖吗
```

**优势**：
- 模型在生成时**不知道**它输出的内容会被拆分。它只是在**自然说话**
- 括号内的内容就是角色"忍不住说出口的内心活动"——有些角色就是会自言自语
- 没有框架约束，模型全程处于"角色体验模式"

**劣势**：
- 后处理提取需要更鲁棒的解析逻辑（括号匹配、语义判断）
- 模型可能不按照格式输出，导致提取失败
- 如果角色性格本身就不爱自言自语，括号内的内容可能很违和

### 4. 混合方案（推荐）

```
Alice 平时很少说私事的，这次肯定真的很累。
——
摸摸头，需要我给你点外卖吗
```

**设计**：
- 用自然分隔符 `——`（破折号）代替标签
- 分隔符**不是指令的一部分**，而是**角色表达方式的一部分**
- 模型看到的是："请回复这条消息。你可以先写一句内心活动，然后写你要说的话，用——分隔。"
- 如果角色不输出内心活动（只回复），后处理检测不到 `——`，就把整段当作 say

**这比 think+say 更自然**——破折号在中国人的日常表达中很常见（"其实我今天不太舒服——但没事，我们继续聊"），模型不会觉得这是一个"系统框架"。

## 实证：不同格式对同一 prompt 的输出差异

假设 prompt 是：
```
你是小暖，一个温暖的大姐姐式角色。
Alice 说："我分手了"
```

**JSON 输出**：
```json
{
  "inner_thought": "Alice 分手了，她现在的情绪应该是悲伤的，我需要给予情感支持。",
  "reply": "抱抱，需要我陪你聊聊吗？"
}
```
→ inner_thought 是**分析报告**（"情绪应该是悲伤的"、"需要给予支持"），不是内心体验。

**think+say 输出**：
```
<think>Alice 分手了... 她平时不太说这些的，这次肯定很难受。我有点心疼她。</think>
<say>抱抱，需要我陪你聊聊吗？</say>
```
→ think 比 JSON 自然一些，但仍有"在写一个内心想法"的刻意感。

**自然语言 + 破折号**：
```
Alice 平时不太说这些的，这次肯定很难受... 我有点心疼她。
——
抱抱，需要我陪你聊聊吗？
```
→ 第一段完全是角色的自然反应，没有"我在填写内心想法"的刻意感。破折号就像角色说完心里话后顿了顿，然后切换到社交模式。

## 推荐方案：破折号分隔 + 可选 fallback

### 主方案：自然分隔符

```python
PROMPT_SUFFIX = """
你可以先写一句你此刻的感受（不会发给任何人），然后写你要回复的内容。
用"——"（两个破折号）分隔感受和回复。

示例：
"""
""""
看到这个词我心里一紧。
——
哈哈，你也太会说了吧
"""
"""
```

**后处理**：
```python
def split_think_say(raw: str) -> tuple[str, str]:
    # 尝试破折号分隔
    if "\n——\n" in raw or "\n——\n" in raw:
        parts = raw.split("\n——\n", 1)
        return parts[0].strip(), parts[1].strip()
    # fallback：整段就是回复，没有内心活动
    return "", raw.strip()
```

### fallback：如果模型没输出分隔符

有些消息不值得角色有内心活动（比如群里的闲聊）。如果模型没有输出 `——`，就把整段当作 say，think 为空字符串。

这是** feature 不是 bug**——角色不需要对每条消息都有内心独白。

### 极端 fallback：如果模型输出了但分隔符位置不对

可以用语义判断：如果 `——` 之前的文本明显是"角色在说话"（包含第一人称、情绪词），就当作 think；否则当作 say 的一部分。

## 一个更激进的方案：完全不分离

**问题**：为什么一定要分离 think 和 say？

**答案**：因为我们要把 think 存入自传体记忆，而 say 发给用户。

**但如果**：
- 我们不分离
- 模型输出的是**完整的角色表达**（包含内心活动和说出口的话混在一起）
- 我们**全部发给用户**
- 同时用**后处理**从完整文本中提取"内心活动"存入记忆

**示例输出**（发给用户）：
```
Alice 平时不太说这些的，这次肯定很难受... 我有点心疼她。
——
抱抱，需要我陪你聊聊吗？
```

用户看到的是带破折号的完整文本。破折号前的部分就是"角色的自言自语"——有些用户可能反而觉得这样更真实、更像一个真人。

**这解决了什么问题**：
- 模型全程处于"自然表达模式"
- 没有"我在写一个系统字段"的框架感
- 破折号可以是角色表达习惯的一部分（有些角色就是爱自言自语）

**这带来了什么问题**：
- 用户可能觉得破折号前的内容"不像是对我说的"
- 需要每个角色的 communication_style 来决定是否使用这种格式

**适用场景**：
- 角色是"话痨型"、"自言自语型" → 适合
- 角色是"高冷型"、"极简型" → 不适合（用户看到多余内容会困惑）

## 结论（用户决策：继续用类 XML 格式）

经过讨论，用户确认：
- **破折号在中文场景下会被误解为角色日常表达**，不适合做分隔符
- **继续沿用类 XML 格式（`<think>` / `<say>`）**

| 格式 | 工程化程度 | 推荐度 | 适用场景 |
|------|-----------|--------|---------|
| JSON | 高 | ❌ | 仅用于分析类任务（情感标注、意图分类） |
| **think+say 标签** | **中** | **✅** | **生产环境（用户确认）** |
| 破折号分隔 | 低 | ⚠️ 已否决 | — |
| 完全不分离 | 最低 | ⚠️ | 特定角色类型（话痨、自言自语） |

**最终建议（已确认）：**
1. 认知层（情感/意图分析）用 **JSON**——这是给系统看的，不是给角色用的
2. 执行层（回复生成）用 **破折号分隔**——这是给角色用的，追求自然感
3. 如果模型没输出破折号，**不要强制分离**——当作角色这次没有内心独白
4. 把是否使用分隔符做成 `PersonaProfile` 的一个字段：`expression_mode = "monologue" | "direct" | "hybrid"`
   - `monologue`：角色会自然地说出内心活动（破折号分隔或不分离）
   - `direct`：角色只输出回复，内心活动由后台异步生成
   - `hybrid`：由模型自己决定（有时有内心独白，有时没有）
