# v0.27.12 迁移说明

本版本继续增强 `desktop_screenshot` 的可用性，重点是让模型在 developer 权限可见时，更稳定地把它当成“判断主机当前正在做什么”的观察工具，而不是普通图片附件。

## 变更摘要

- `desktop_screenshot` 的技能描述已改为显式面向“判断主机当前在做什么、屏幕上打开了哪些窗口/应用/页面”的场景。
- 当当前轮次可见 `desktop_screenshot` 时，SKILL 系统提示会明确提醒模型：若要判断主机当前状态，应先截图再回答，而不是直接猜测。
- `desktop_screenshot` 新增可选参数 `focus`，可用于指定本次观察重点，例如“判断主机当前在做什么”“确认前台窗口”“查看当前页面”。
- 截图返回的内部文本现在会强调观察前台窗口、可见应用、页面标题、编辑器/终端内容；如果仅凭截图无法确认后台任务，需要明确说明不确定。

## 对外影响

### 1. 模型更容易主动调用 `desktop_screenshot`

如果当前发言者具备 developer 权限，且问题本身是在询问：

- 主机当前在做什么
- 屏幕上现在是什么
- 打开了哪些窗口、应用或页面
- 当前前台工作内容是什么

模型现在更容易先调用 `desktop_screenshot`，再基于截图回答。

这不是新的权限放宽：

- 非 developer 当前轮次仍然看不到 `desktop_screenshot`。
- 即使模型强行伪造调用，runtime 仍会拒绝。

### 2. 可选 `focus` 参数

现在可以这样调用：

```python
[SKILL_CALL: desktop_screenshot | {"focus": "判断主机当前在做什么"}]
```

或：

```python
[SKILL_CALL: desktop_screenshot | {"focus": "确认前台窗口和当前页面"}]
```

如果不传，内置默认重点会回退为“判断主机当前在做什么，以及前台窗口、应用、页面和任务状态”。

### 3. 截图结果的解释边界更严格

截图返回后，模型会收到更明确的内部提示：

- 优先根据前台窗口、可见应用、页面标题和编辑器/终端内容判断当前状态。
- 对截图能看到的内容做结论。
- 对截图看不到的后台动作或隐含状态，不要臆测。

## 升级检查清单

升级到 v0.27.12 后，建议至少验证以下场景：

1. developer 当前轮次下，询问“电脑现在在干什么”时，模型能优先调用 `desktop_screenshot`。
2. 截图返回后，模型会围绕前台窗口/页面/应用给出结论，而不是把截图仅当成普通图片。
3. 传入 `focus` 参数时，结果中的内部文本和 capture 历史会记录该重点。
4. 非 developer 当前轮次下，`desktop_screenshot` 仍然不会暴露给模型。
