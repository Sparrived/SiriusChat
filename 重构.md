# 全局要求：
 1. 现在为重构代码，删除所有已经过时的代码，旧系统的代码，以及相关测试。
 2. 重构代码必须满足以下要求：
    - 代码必须符合最新的设计规范和编码标准。
    - 代码必须经过充分的测试，确保其功能正确且稳定。
    - 代码必须具有良好的可读性和可维护性，便于后续的维护和扩展。
    - 代码必须经过代码审查，确保其质量和安全性。
 3. 每完成一个小的重构步骤就提交一次commit
 4. 为这次重构创建一个新的分支，命名为“refactor-memory-system”，并在完成重构后将其合并到主分支。
 5. 确保文档的时效性和准确性，更新相关的设计文档、API文档和用户手册，以反映新的系统设计和功能。

# 对记忆系统的问题：
 1. 当前事件系统过于繁琐，并且实际体验不佳。说是多层记忆，但实际上都是简单的算法赋值，没有语义上的递进体现。
 2. 与用户系统的功能重合

# 对用户系统的问题：
    1. 当前用户系统过于复杂，包含了许多不必要的功能和模块，导致代码冗长且难以维护。
    2. 用户系统实际上有很多字段是没有实际意义的，或者说是过于冗余的，从头至尾几乎没有被使用。这些字段应该被删除或者合并，以简化系统的设计和实现。

# 对框架的问题：
    1. 当前架构太依赖外部（例如插件侧）的实现了，导致插件实际上与框架的耦合度过高，无法独立开发和测试。

# 调整方案
 1. 直接推翻当前的记忆系统，我建议的新记忆系统方案：
    - 记忆系统应该分为两层：第一层是基础记忆（即构建简单的上下文记忆），第二层是日记记忆。
    - 基础记忆将最近的消息全盘保留（同时持久化）作为上下文传给模型，可以设置一个硬边界（例如30条消息），到达这个上限也进入日记记忆的测算。
    - 通过计算群聊热度，当群聊冷淡时将全盘的基础记忆进入日记LLM，同时为LLM传入人格，确保LLM生成的内容符合人格设定。
    - 将记录的日记通过RAG嵌入语义向量库。
    - 基础记忆去掉已经在日记记忆记录的消息，保留最近的3-5条上下文，继续参与后续的对话生成。
    - 为了方便模型快速的自主学习，名词解释功能可以保留，但需要确保其内容能够被有效地检索和利用，以支持后续的对话生成和用户交互。
 2. 关键问题
    - 需要确保用户输入、模型输出以及系统提示词的全量落盘归档，确保数据的完整性和可追溯性。
    - 需要额外的后台算法对记录的日记进行归纳总结与去重，确保日记的质量和实用性。
    - 需要设计一个合理的算法来计算群聊的热度，以便在适当的时候将基础记忆转移到日记记忆中。
    - 需要确保日记记忆的内容能够被有效地检索和利用，以支持后续的对话生成和用户交互。
    - 可以给定后续上下文可传输的日记数量，确保不会token爆炸，同时也能保证日记的实用性和相关性。
  3. 用户系统调整方案：
    - 删除所有过于复杂和冗余的功能和模块，保留核心的用户信息和功能。
    - 合并或者删除没有实际意义的字段，简化用户系统的设计和实现。
    - 确保用户系统的设计符合最新的需求和规范，同时也具有良好的可维护性和可扩展性。
    - 探索用户系统与新的记忆系统的集成方式，确保两者能够有效地协同工作，提升整体系统的性能和用户体验。
  4. 框架调整方案：
    - 重新设计框架的架构，减少对外部实现的依赖，确保框架能够独立开发和测试。
